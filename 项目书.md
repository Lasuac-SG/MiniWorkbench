1. 前置环境依赖

C++ 标准： C++20。

构建系统： CMake 3.16及以上。

核心框架： Qt 6 包含 Core, Gui, Qml, Quick, Sql, Concurrent 模块。

第三方库： libgit2。通过 CMake 引入，禁止硬编码本地绝对路径。

目标平台： 跨平台。文件路径处理使用 QDir 或 std::filesystem，严禁硬编码分隔符。

2. 项目目标

开发常驻桌面的模块化数据看板。

核心形态： 启动即为无边框、背景透明窗口。默认隐藏在屏幕边缘，支持鼠标悬停 0.5 秒后平滑滑出。

UI 范式： 提供画布，允许通过配置动态加载、拖拽、缩放多个组件。

业务领域： 提供 TodoList 任务管理、基于本地 Git 仓库轮询的代码贡献热力图。

3. 核心开发要求

3.1 强制设计原则与规范

抽象： 精确定义接口边界，对外仅暴露必要交互契约。

DRY： 严禁代码复制粘贴，重复逻辑必须提取。

KISS： 拒绝过度设计，选择最简单直接的实现路径。

YAGNI： 仅在需要时添加代码，严禁编写冗余抽象层。

LoD： 严格遵守交互限制，对象只与其直接依赖项通信，禁止链式调用暴露内部结构。

高内聚低耦合： 相同职责高度集中；UI 渲染层与业务数据逻辑层必须物理隔离与单向依赖。

代码精简： 拒绝冗余设计，善用 C++20 语法糖。

注释详细： 类声明与复杂算法逻辑必须包含 Doxygen 风格注释说明时间复杂度和内存所有权。

面向对象与封装化： 严格控制访问权限。禁止暴露不必要内部指针，C API 必须完全封装在 C++ 类中。

3.2 核心规范约束

MVC架构： QML 仅负责渲染和动画。业务逻辑、文件读写、Git 轮询必须在 C++ 中执行。C++ 与 QML 的列表数据传递强制使用继承自 QAbstractListModel 的方式。

防御性编程与 RAII 资源管理： 禁止使用裸指针分配内存。强制使用 std::unique_ptr 或 QScopedPointer。封装 libgit2 时必须使用自定义 Deleter 的智能指针接管资源释放。

线程安全与熔断： Git 历史遍历必须在独立子线程中执行。向主线程传递数据必须使用 Qt 信号与槽。

## 5. 已完成部分与上下文交接 (截至 Block 5 结束)

本段落作为 AI 代理跨 Session 记忆交接的依据。当前项目已完成 Block 1 至 Block 5，并全部编译通过。

### 5.1 核心需求与架构约束重申
- **技术栈**：C++20, Qt6 (Core/Gui/Qml/Quick/Sql/Concurrent), CMake 3.16+, libgit2。
- **架构模式**：MVC。QML **仅** 负责渲染和动画，业务逻辑与数据模型完全在 C++。
- **通信约束**：QML 与 C++ 的数据列表传递必须使用 `QAbstractListModel`。QML 不能直接修改底层存储，必须通过 `Q_INVOKABLE` 或槽函数发出修改请求。

### 5.2 Block 1: 基础设施与窗口管理层 (已完成)
- **`CMakeLists.txt`**：已配置 Qt6 依赖与 `libgit2`，并启用了自动 MOC/UIC/RCC。
- **`WindowManager.h/cpp`**：实现了控制底层原生窗口属性的类，配置了 `Qt::FramelessWindowHint | Qt::WindowStaysOnTopHint | Qt::Tool` 及 `Qt::transparent` 背景，暴漏给了 QML 层。
- **`Main.qml`**：构建了**信息胶囊**与**展开面板**两种状态的平滑切换基础。

### 5.3 Block 2: 动态微件系统基础 (已完成)
- **`widget_config_model.h/cpp`**：基于 `QAbstractListModel` 实现了 `WidgetConfigModel`，暴露了微件的核心结构 `Roles`。
- **`GridLayoutEngine.qml`**：读取 `WidgetConfigModel` 渲染具体的 Widget 矩形方块。处理拖拽 (`DragHandler`) 和缩放 (`MouseArea`)。

### 5.4 Block 3: SQLite 本地存储持久化 (已完成)
- **`database_manager.h/cpp`**：实现了 `DatabaseManager` 单例，在启动时连接 `config.db`，并提供加载和修改数据库的 CRUD 接口。
- **`WidgetConfigModel` 改造**：初始化读取数据库，所有添加、删除、更新操作自动落盘保存。

### 5.5 Block 4: 高性能 Git 解析后端 (已完成)
- **`git_analyzer.h/cpp`**：
  - 实现了 `GitAnalyzer`，封装了 `libgit2` 以提取代码增删行数。
  - **内存安全**：为所有 libgit2 原生对象 (`git_repository`, `git_revwalk`, `git_commit`, `git_tree`, `git_diff`) 提供带自定义 Deleter 的 `std::unique_ptr` 封装（如 `GitRepoPtr`, `GitRevwalkPtr` 等），确保 RAII 资源管理。
  - **异步与线程安全**：通过 `QtConcurrent::run` 将遍历仓库等重度操作放入独立子线程执行。执行完成后利用 `QFutureWatcher` 和 Qt 信号 (`analysisFinished`) 安全将数据传回主线程。
  - **后缀名过滤**：在 `git_diff_foreach` 提供的回调函数中匹配后缀名，精准提取对应文件的增加和删除行数。

### 5.6 Block 5: 热力图渲染层 (已完成)
- **`heatmap_item.h/cpp`**：
  - 实现了继承自 `QQuickPaintedItem` 的自定义 QML 绘制项 `HeatmapItem`，并通过 `QML_ELEMENT` 注册到 QML 引擎。
  - **高性能绘制**：重写了 `paint` 函数，直接利用底层的 `QPainter` 批量高效无损重绘方块热力图，弃用了 QML 层面的 DOM 节点堆叠。内置了一套代码提交活跃度的颜色深浅计算逻辑。
- **视图联调 (`GridLayoutEngine.qml`)**：
  - 根据微件类型动态加载 `HeatmapItem` 组件。
  - 对外暴露了 `GitAnalyzer` 的 QML 接口，能够由 QML 侧驱动（`startAnalysis`），并将回调数据与 `HeatmapItem` 绑定。

### 5.7 下一步开发目标: Block 6 (TodoList 微件与最终联动)
